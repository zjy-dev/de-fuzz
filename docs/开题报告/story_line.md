1. 背景与问题的严重性

我们正处于"漏洞过载"的年代——CVE 数量逐年攀升, 工业界已经意识到上层代码的 Bug 不可能被彻底修干净。正因为漏洞终将存在, 防御机制的有效性比以往任何时候都更关键: 编译器与运行时叠加的防御机制 (栈金丝雀、CFI、IBT、PAC/BTI 等) 成为系统安全的"兜底层"——它们的设计初衷就是在漏洞被触发时, 仍能挡住攻击者拿到控制权。

然而, 这道兜底防线本身也会出错。防御机制的有效性除了依赖自身代码实现, 还与底层指令集架构 (ISA) 的假设强相关。比如 GCC 在 AArch64 上的 -fstack-protector 长期失效 (CVE-2023-4039): 当函数使用变长数组时, 编译器把金丝雀放在 VLA 之上、返回地址却落在 VLA 之下, 越界写入可以先于金丝雀检查改写返回地址——一个本应"兜底"的机制在该架构上形同虚设。当防御设计和特定架构的假设产生冲突时, 就会出现这类失效。

这种失效具有极具破坏性的"双重无声"特征: 编译时不报错, 运行时功能也完全正常 (二进制无声), 唯独防御的有效性已经被打破。

2. 核心挑战: 搜索空间巨大且失效难以判定

检测此类失效面临两个相互纠缠的挑战:

搜索空间巨大。防御机制种类繁多 (金丝雀、_FORTIFY_SOURCE、CET/IBT、CFI、BTI、PAC、Shadow Stack 等), 每种机制在 x86-64、AArch64、RISC-V、LoongArch 等多套 ISA 上又各有独立的后端实现。"机制 × ISA"的二维矩阵规模庞大, 人工审计难以覆盖。
失效难以判定 (oracle 问题)。即使把执行推进到了可疑代码路径, 如何判断"防御已被静默削弱"也并不直接——程序可能正常运行、不崩溃、与其他编译器输出也一致, 但安全契约其实已经失守。这是此类 Bug 与传统编译器 Bug 最本质的区别。
3. 现有方法的局限  

面对上述双重挑战, 现有的自动化分析手段都遇到了瓶颈:

静态分析依赖已知的代码模式。但防御机制失效的形态千奇百怪, 大部分案例都有独特的违例方式, 靠模式匹配难以覆盖。
模糊测试 (Fuzzing) 的传统预言机往往只盯着"程序是否崩溃"或"与其他编译器输出是否一致", 这些表面指标从本质上无法揪出防御机制的静默逻辑失效——也就是上节提到的 oracle 空白。
4. 我们的解决方案

为了同时应对庞大的搜索空间与 oracle 难题, 我们设计了一套专门的预言机机制:

从文档与代码注释中, 梳理出每项机制必须遵守的核心"安全不变量", 也即一套"安全规则" (例如: 金丝雀必须位于所有可被越界写入的栈对象与返回地址之间), 并将其转化为可由机器验证的静态属性 (看汇编/二进制特征) 与动态属性 (看运行时行为)。
把这些规则用程序实现为具体的检查器 (checker), 其中相当一部分规则跨 ISA 通用, 可在不同后端复用。
将检查器合理编排, 拼装成针对该防御机制的专属预言机, 直接回答"防御是否仍然有效"——填补 oracle 空白。
结合 LLM 与覆盖率驱动的 Fuzzing 生成高多样性种子, 自动深入防御机制相关代码路径, 逐格排查"机制 × ISA"矩阵。
