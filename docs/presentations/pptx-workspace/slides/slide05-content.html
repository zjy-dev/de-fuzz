2. 研究内容 - 核心挑战与解决思路

编译器软件防御机制模糊测试面临的关键难题

挑战1: 测试预言机缺失

防御失效是"静默"的逻辑漏洞，无崩溃、无报错，传统 Fuzzer 的 crash oracle 完全失效

->

方案: 插件化 Oracle 体系

针对不同防御策略设计专用检测逻辑（如 Canary 二分搜索、FORTIFY 边界验证），LLM Oracle 作为通用后备

挑战2: 深层路径难触发

防御代码往往位于编译器复杂的条件分支深处，随机变异难以生成满足路径约束的输入

->

方案: LLM 约束求解

利用 LLM 代码理解能力，基于 CFG 选择高价值目标，渐进式生成满足约束的测试用例

挑战3: GCC 静态分析困难

GCC 代码库超千万行，防御代码分散多模块，缺乏现成的覆盖率和 CFG 提取工具

->

方案: 定制化插桩编译器

构建覆盖率插桩的 GCC，提取目标函数 CFG，支持多架构交叉编译

挑战4: 跨架构测试复杂

不同 ISA 的栈布局、调用约定差异大，漏洞可能仅在特定架构触发

->

方案: QEMU + 统一抽象

通过 QEMU 执行交叉编译产物，设计架构无关的 Oracle 接口
