1. 断点续传功能在info级别日志中没有明确的显示开始和结束, 且断点续传应该和只在 initial_seeds 目录下有种子的情况区分开(后者应该明确提示无之前的运行结果, 只有初始种子而已)
2. 目前的 fuzzer 有点慢, 感觉不光是 LLM 的问题, 我们的覆盖率命令/oracle等传统 fuzzer 部分的性能好像也比较低, 我们应该创建一个专门的集成版压力测试, 复制一个种子例如64次, 然后设置 --limit 0 来只跑断点续传的部分来明确传统部分的性能. 我们还需要在 debug 级别添加不同组件的运行时间日志, 方便我们明确瓶颈在哪. 在此之前我们需要注释掉一些prompt相关的debug log
3. 观察 fuzz_out 目录, 会发现 metadata 里存的种子的覆盖率都是对的, 但是 corpus 里的目录命名中 cov 全是 0, 显然不对劲.
4. 大模型的超参数应该配置的更像为 fuzzer 服务, 例如 temperature 应该非常低, 其他的超参数我不太懂你看着调
